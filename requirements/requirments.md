# 背景
1. 用户主要使用 cp2k 和 vasp 进行第一性原理计算。
2. 服务器[环境](./env.md)使用 [PBS](./PBS.md) 管理作业调度。
3. 提交任务的脚本范例参考[示例](./scripts_template/server1)中的内容。
4. 用户可用 240 核，单次提交任务仅允许提交 24 核的整数倍。
5. 用户无 sudo 或 root 权限。

# 目标
1. 使用 Python 实现一个大批量自动提交 PBS 任务的工具。
2. 工具需要支持状态保持（例如一批 100 个任务的提交进度、状态和时间耗费可持续记录）。
3. 可视化形式为 CLI 界面。
4. 工具需要支持 cp2k 与 vasp 两类任务目录的批量提交。

# 权限约束
1. 工具必须在普通用户权限下运行，不依赖 sudo 或 root。
2. 工具不进行系统级安装、系统配置修改或需要提权的操作。

# 输入目录与任务约定
1. 大批量任务位于某个根目录下的多个子目录中。
2. 默认每个一级子目录都视为有效任务目录，不做额外有效性筛选。
3. 任务提交名取子目录名称。
4. 示例目录参考 `wordir_example`。

# 提交规则
1. 默认在任务子目录中执行提交命令：`qsub -N <子目录名> script.sh`。
2. 任务状态获取依赖 PBS 查询命令（`qstat` 或 `q`）。
3. 成功提交的任务状态样板参考 [successfully_submitted.md](./successfully_submitted.md)。

# 资源与限流规则
1. 运行中任务（`R`）的总核数上限为 240。
2. 队列等待任务（`Q`）允许的总核数上限为 192（即 `96 * 2`）。
3. 队列等待核数不计入运行中核数统计。
4. 每个任务核数必须为 24 的整数倍（从提交脚本资源请求中读取）。
5. 提交器在每次新提交前都需要检查 `R` 和 `Q` 两个限额，任一达到上限时暂停新提交并轮询。

# 状态判定规则
1. `qsub` 返回成功并拿到 `JOBID` 后，任务状态记为 `submitted`。
2. 在 `submitted` 后 60 秒执行一次 `qstat`/`q` 检查：
   - 若任务处于计算中（`R`），判定为成功提交；
   - 若仍在排队（`Q`），保持等待状态，不判定为失败。
3. 若任务在开始计算后 30 秒内结束，输出 `warning` 提示。
4. 若 `qsub` 命令失败（非 0 返回或无 `JOBID`），任务记为提交失败。

# 持久化与CLI展示
1. 工具需要在批量执行过程中持续保存任务状态，支持中断后继续。
2. CLI 至少展示：总任务数、已提交数、等待中数、运行中数、成功提交数、提交失败数、warning 数。
3. CLI 需要展示本批次累计耗时和最近状态变更时间。
